## 1. システム概要・目的

### 1.1 システム概要

本システムは、営業担当者が訪問先企業に対してセキュリティレベルの診断を行い、想定被害額を可視化したうえで商材提案を行うためのWebアプリケーションである。PHPで構築されたシンプルな多ページ型Webアプリとして設計する。

### 1.2 システムの目的

| 目的 | 詳細 |
|------|------|
| セキュリティ課題の可視化 | 訪問先企業のセキュリティ上の弱点を定量的なスコアで示す |
| リスクの金銭的換算 | 業種・規模・スコアをもとにサイバー攻撃による想定被害額を算出する |
| 商材への自然な導線 | 診断結果から、DDHBOXおよびnecfru MAM/DAMの必要性を論理的に説明する |
| 営業活動の標準化 | 担当者のスキルに依存しない、一定品質の提案を実現する |

### 1.3 販売対象商材

| 商材 | 提供会社 | 位置づけ |
|------|----------|----------|
| DDHBOX | デジタルデータソリューション株式会社 | ネットワーク型不正通信遮断ハードウェア（主力商材） |
| necfru MAM/DAM | 株式会社ネクフル | クラウドストレージ・データ管理SaaS（補完商材） |

## 2. ターゲットユーザーとユースケース

### 2.1 ターゲットユーザー

#### 主ユーザー：営業担当者

- 本システムを操作し、顧客に画面を見せながら診断を進める
- ITセキュリティの専門知識は必須ではない
- タブレットまたはノートPCを使って訪問先で操作する

#### 閲覧対象：訪問先企業の担当者

- 情報システム担当者、総務・経理担当者、経営層など
- セキュリティ知識は多様（詳しい人から初心者まで）
- 難解な専門用語を避け、金額やリスクで直感的に理解できる表現が必要

### 2.2 ユースケース一覧

| UC番号 | ユースケース名 | アクター | 概要 |
|--------|---------------|----------|------|
| UC-01 | 企業情報入力 | 営業担当者 | 訪問先企業の基本情報（業種・従業員数・PC台数等）を入力する |
| UC-02 | セキュリティチェック実施 | 営業担当者（顧客と一緒に） | チェックリスト形式で現状のセキュリティ対策状況を入力する |
| UC-03 | 診断結果確認 | 営業担当者・顧客担当者 | セキュリティスコア・リスクレベル・想定被害額を確認する |
| UC-04 | 商材提案確認 | 営業担当者・顧客担当者 | 診断結果に基づく推奨商材・プランを確認する |
| UC-05 | 診断リセット | 営業担当者 | 次の顧客のために入力内容をリセットする |
| UC-06 | 結果出力（将来機能） | 営業担当者 | 診断結果をPDFとして出力・送付する |

### 2.3 利用シナリオ（標準フロー）

```
営業訪問開始
  └─ (1) index.php で企業情報を入力
       └─ (2) assessment.php でチェックリストに答える
            └─ (3) result.php で診断結果・提案を顧客と一緒に確認
                 └─ (4) 商談へ移行（DDHBOX・necfru MAM/DAM）
```

## 4. 非機能要件

### 4.1 パフォーマンス

| 項目 | 要件 |
|------|------|
| ページ応答時間 | 各画面の初期表示を2秒以内とする |
| 同時接続 | 営業用途のため同時接続数は最大10セッション程度を想定 |
| ネットワーク | オフライン環境（訪問先Wi-Fiなし）でも動作できるようローカル実行を考慮する |

### 4.2 セキュリティ

| 項目 | 要件 |
|------|------|
| 入力値サニタイズ | XSSを防ぐためすべての出力値をhtmlspecialcharsで処理する |
| CSRFトークン | フォーム送信にCSRFトークンを付与する |
| セッション管理 | 認証後にsession_regenerate_id()を実行する |
| ファイルアクセス制御 | includesディレクトリへの直接HTTPアクセスを.htaccessで禁止する |
| HTTPS | 本番環境ではHTTPS接続を必須とする |

### 4.3 可用性

| 項目 | 要件 |
|------|------|
| ブラウザ対応 | Chrome・Edge・Safariの最新2バージョンに対応する |
| レスポンシブ | タブレット（iPad等）での表示を担保する |

### 4.4 保守性

| 項目 | 要件 |
|------|------|
| ロジックの分離 | 料金・スコア計算はcalculator.phpに集約し、画面ロジックと分離する |
| 設定の外部化 | 料金プランや係数などの設定値はconfig.phpに集約する |
| コメント | 計算ロジックには係数の根拠を日本語コメントで記述する |

**チェックカテゴリと配点**

| カテゴリID | カテゴリ名 | 配点（満点） | 設問数 |
|------------|-----------|-------------|--------|
| CAT-01 | ネットワーク・境界防御 | 25点 | 5問 |
| CAT-02 | エンドポイント対策 | 20点 | 4問 |
| CAT-03 | データ管理・バックアップ | 20点 | 4問 |
| CAT-04 | アクセス管理・認証 | 20点 | 4問 |
| CAT-05 | インシデント対応・教育 | 15点 | 3問 |
| **合計** | | **100点** | **20問** |

**回答形式**

- 「はい（対策済み）」「わからない（部分実施）」「いいえ（未対策）」の3択
- 「わからない」は各設問の50%を付与

**表示ブロック構成**

```
[A] 企業情報サマリー
[B] セキュリティスコア（円グラフ or ゲージ）＋ カテゴリ別レーダーチャート
[C] 想定被害額（最小〜最大レンジ表示）
[D] DDHBOX提案ブロック（推奨プラン・月額・サイバー保険訴求）
[E] necfru MAM/DAM提案ブロック（費用試算・主要機能）
[F] 費用対効果サマリー（被害額 vs 導入費用の比較）
[G] アクションボタン（リセット・PDF出力・資料DL）
```

**リスクレベル定義**

| スコア範囲 | リスクレベル | 表示色 | メッセージ |
|-----------|-------------|--------|-----------|
| 0〜39点 | 危険 | 赤 | 早急な対策が必要です |
| 40〜59点 | 警戒 | オレンジ | 重要な対策が不足しています |
| 60〜79点 | 注意 | 黄 | 一部に改善が必要です |
| 80〜100点 | 良好 | 緑 | 基本的な対策は整っています |

## 7. 計算ロジック設計

### 7.1 セキュリティスコア算出

**回答ごとの点数**

| 回答 | 換算率 |
|------|--------|
| はい（対策済み） | 100% |
| わからない（部分実施） | 50% |
| いいえ（未対策） | 0% |

**カテゴリスコア算出式**
```
カテゴリスコア = (獲得点数 / カテゴリ満点) × カテゴリ配点
総合スコア = 全カテゴリスコアの合計
```

**調整係数**

| 条件 | 調整 |
|------|------|
| 個人情報を取り扱う | -5点（下限0） |
| 従業員数500名以上 | -3点（下限0） |

### 7.2 想定被害額算出

**規模別基準被害額**

| 従業員数 | 基準被害額（万円） |
|---------|-----------------|
| 1〜29名 | 500 |
| 30〜99名 | 1,500 |
| 100〜299名 | 5,000 |
| 300〜999名 | 15,000 |
| 1,000名以上 | 50,000 |

**スコア係数**

| スコア範囲 | 係数 |
|-----------|------|
| 0〜39点 | 2.0 |
| 40〜59点 | 1.5 |
| 60〜79点 | 1.1 |
| 80〜100点 | 0.8 |

**算出式**
```
中央値被害額 = 基準被害額 × 業種リスク係数 × スコア係数
最小被害額 = 中央値 × 0.6（切り捨て）
最大被害額 = 中央値 × 1.6（切り上げ）
※個人情報取扱あり：最大被害額 +1,000万円
```

### 7.3 DDHBOX推奨プラン算出

**プラン定義テーブル**

| プラン名 | 対応台数上限 | 月額（円） | 年額（円） |
|---------|------------|-----------|-----------|
| プラン10 | 10台 | 23,900 | 286,800 |
| プラン20 | 20台 | 29,400 | 352,800 |
| プラン30 | 30台 | 40,700 | 488,400 |
| プラン50 | 50台 | 52,000 | 624,000 |
| プラン100 | 100台 | 74,700 | 896,400 |
| プラン300 | 300台 | 183,300 | 2,199,600 |
| プラン500 | 500台 | 305,500 | 3,666,000 |
| プラン1000 | 1,000台 | 611,000 | 7,332,000 |

**選定ロジック**
```
推奨プラン = PC台数 <= 対応台数上限 となるプランのうち月額最小のもの
例: PC25台 → プラン30（30台対応）を推奨
```

## 8. 商材提案ロジック

### 8.1 提案強度の定義

| 提案強度 | 表示スタイル |
|---------|------------|
| 強く推奨 | 赤枠・強調表示 |
| 推奨 | 青枠・通常表示 |
| 参考提案 | グレー・軽微表示 |

### 8.2 DDHBOXの提案条件

| 条件 | 提案強度 |
|------|---------|
| 総合スコア 0〜39点 | 強く推奨 |
| 総合スコア 40〜59点 | 強く推奨 |
| CAT-01スコアが60%未満 | 強く推奨 |
| 総合スコア 60〜79点 | 推奨 |
| 総合スコア 80〜100点 | 参考提案（保険付帯訴求） |

### 8.3 necfru MAM/DAMの提案条件

| 条件 | 提案強度 |
|------|---------|
| CAT-03スコアが50%未満 | 強く推奨 |
| CAT-03スコアが50〜79% | 推奨 |
| 個人情報取扱あり かつ CAT-04スコアが70%未満 | 強く推奨 |
| 総合スコア 80〜100点 | 参考提案（BCP対策訴求） |

### 8.4 費用対効果の表示

```
年間導入費用 = DDHBOX年額 + necfru月額 × 12
費用対効果倍率 = 想定被害額（最小） / 年間導入費用

表示例:
「想定被害額（最小 〇〇万円）に対し、年間導入費用は〇〇万円です。
 被害額の〇〇倍以上のリスクに備えることができます。」
```

## 付録：設計上の決定事項と根拠

| 決定事項 | 選択 | 根拠 |
|---------|------|------|
| フレームワーク | 不使用（現行踏襲） | 既存コードとの互換性・シンプルな用途 |
| データ永続化 | セッションのみ（フェーズ1） | 追加インフラ不要。管理画面実装時にDB追加 |
| チャート描画 | Chart.js（CDN利用） | 軽量・PHP相性・ドキュメント豊富 |
| スコア計算方式 | 加点方式 | 直感的で説明しやすい |
| 被害額の表示方式 | レンジ表示（最小〜最大） | 不確実性を正直に伝えられる |
| 設定値管理 | config.php集約 | 料金改定・係数調整時の変更箇所を1ファイルに限定 |
